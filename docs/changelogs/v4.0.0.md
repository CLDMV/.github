# v4.0.0 — 2026-05-22

Major release cutting over from the v3 per-PR release flow to a staging-branch model. The new flow uses long-running `next` and `hotfixes` integration branches plus persistent `<branch> → master` release PRs that batch multiple feature/fix commits into a single release. The architectural mechanism was built incrementally during v3.3–v3.8 (foundation actions, hotfix lane, pending-release reminder, README/bootstrap/migration guide); v4.0.0 is the activation cut — retiring the v3 fan-out and bumping the version line.

See also: [migration guide for v3 → v4 consumers](../migration/v3-to-v4.md) and the full design in [conventions/release-flow-v4.md](../conventions/release-flow-v4.md).

## Breaking changes

- **Release flow restructured: per-PR fan-out → staging-branch model.** v3 attached a `release: vX.Y.Z` PR to every release-eligible feature/fix PR. v4 collects all feature/fix work onto two long-running integration branches (`next` for features + non-urgent fixes, `hotfixes` for urgent patches), each of which has a single persistent release PR back to `master`. Master only changes via a reviewed squash of one of those persistent PRs. Consumer impact: contributors merge into `next` (or `hotfixes`) instead of `master`; release timing decouples from individual PR merges. (`feat!:` 5fbaaa4)
- **All consumer workflow refs must move from `@v3` to `@v4`.** The `@v3` major tag stays pinned to the last v3.x release (immutable, unmaintained, available indefinitely). Internal repo references are updated repo-wide; consumer templates in `examples/individual-repo-workflows/` point at `@v4`.

## Added

### Staging-branch flow workflows

- **`local-next-release.yml`** — fires on push to `next`; resolves-or-creates the persistent `next → master` release PR. (Built in v3.5 as `next-release.yml`; activated in v4.)
- **`local-hotfixes-release.yml`** — mirror for the `hotfixes` lane. (Built in v3.6; activated in v4.)
- **`local-next-reset.yml`** — after each release lands on master, force-resets `next` (and `hotfixes`) to master HEAD via the Git Refs API. Includes a **wait-for-tags** job that polls the released major (`@vN`, parsed from the `release:` commit) until the tag matches the release commit before resetting — prevents the "ship action fix in release N, but release N runs with the OLD action code" race.
- **`local-hotfix-redirector.yml`** — redirects PRs incorrectly targeting `master` to the appropriate integration branch.
- **`local-pr-title-normalizer.yml`** — normalizes PR titles to the conventional-commits format expected by the release-PR machinery. (Built in v3.4 as PR 2; activated in v4.)
- **`local-v4-bootstrap.yml`** — one-shot creator for `next` and `hotfixes` branches when adopting v4 in an existing repo.
- **`local-pending-release-reminder.yml`** — periodic reminder if the persistent release PR has been open too long without a cut. (Built in v3.7; activated in v4.)

### Release-flow actions

- **`actions/github/jobs/create-release-pr`** + **`update-release-pr`** — the core machinery that creates / refreshes the persistent release PR, computes the next version from the commit range, populates the changelog, and pushes the `chore: bump version` commit to the integration branch.
- **`actions/github/jobs/force-reset-branch`** — resets an integration branch to a target SHA. Resets via `PATCH /git/refs/heads/<branch>` (`force: true`) on the REST API; falls back to `git push --force-with-lease=ref:sha` (CLI) with the lease read via `ls-remote`. A bare `--force-with-lease` fails against `x-access-token` URLs because there's no remote-tracking ref to compare against. (Built in v3.5; refined through v3.8.1.)
- **`actions/git/steps/check-release-commit`** — recognizes `release:` commits so the next/reset machinery treats them correctly (don't loop, don't re-bump, don't churn labels).
- **`actions/git/steps/compute-highest-commit-type`** — foundation action used by the release-PR machinery to derive the next version from the accumulated commits in the integration branch. (Built in v3.3 as PR 1.)

### Documentation and rulesets

- **[`docs/conventions/release-flow-v4.md`](../conventions/release-flow-v4.md)** — the design spec for the v4 flow: branch model, release PR lifecycle, hotfix lane, reset semantics, bot bypass requirements, and the institutional gotchas. The single source of truth for "why v4 works this way."
- **[`docs/migration/v3-to-v4.md`](../migration/v3-to-v4.md)** — the operational checklist for cutting a repo over, plus the rollout gotchas (bot App bypass behavior, hotfix vs hotfixes naming, wait-for-tags requirement, App-bypass-on-API-but-not-git-push, version bump on integration branch).
- **`data/rulesets/master.json` / `next.json` / `hotfixes.json`** — exportable rulesets to enforce the v4 flow's branch-protection requirements (require PR + block force-push on protected branches; bot App in the `next` / `hotfixes` bypass list).

## Removed

- **`workflow-sync-open-release-prs.yml`** + **`local-sync-release-prs.yml`** — the v3 fan-out workflow that refreshed every open release PR when master moved. v4 has one persistent PR per lane, so there's nothing to fan out.

## Patch releases summary

Per the [README convention](README.md), patch releases inside this minor are summarized here rather than getting their own files.

### v4.0.1 — 2026-05-22 (PR #27)

Fixes uncovered during the first hours of operating v4 in production.

- **Keep v4 integration branches alive across releases** — the post-release reset path could leave `next`/`hotfixes` in a state that broke subsequent operations; reset semantics revised. (c7dca2d)
- **Stop release-PR label churn on refresh** — every refresh was re-applying labels even when unchanged, producing noisy timelines. Refresh now only edits labels that actually need changing. (420d914)
- Hardened `check-release-commit` semantics with test coverage; refined `local-next-reset.yml` and `local-v4-bootstrap.yml` flows to handle edge cases (empty integration branch, race between reset and tag rollover).

### v4.0.2 — 2026-05-22 (PR #28)

- **Point this repo's own refs at `@v4`** — activates v4.0.1's fixes for the repo's own dogfood workflows. v4.0.1 added the code but didn't roll the repo's internal `uses: …@v3` references to `@v4`. This release does the swap repo-wide so the dogfood actually exercises the v4 path. (03bd6f8)
