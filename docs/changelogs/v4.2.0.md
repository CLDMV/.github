# v4.2.0 — 2026-05-23

Minor release covering consumer-facing v4 adoption (release-flow-v4 templates packaged for copy-paste consumers, v4-first scaffolding guide updates) plus a cluster of fixes uncovered while bringing the CLDMV portfolio onto the v4 flow.

## Added

- **`examples/individual-repo-workflows/release-flow-v4/` directory** — six copy-paste templates that consumers drop into their `.github/workflows/` to opt into the v4 flow: `next-release.yml`, `hotfixes-release.yml`, `next-reset.yml`, `hotfix-redirector.yml`, `pr-title-normalizer.yml`, `v4-bootstrap.yml`. Each template has placeholder `package-name` / `build-command` values plus inline comments explaining what to customize per repo. (feat: 327b011)
- **`examples/guides/AGENT-SCAFFOLDING.md` substantially expanded** — the scaffolding guide now leads with v4 conventions: which workflows to install for a new repo, which to skip, how the integration branches map to release lanes, and the order of operations for first-time adoption.

## Changed

- **`README.md` reframed for v4 as the current line.** Drops references to the now-deleted v3 fan-out workflows. `@v4` documented as current; `@v3` documented as legacy / frozen. Removed bullets and screenshots that no longer reflect the actual repo state. (fix: 1b523ba)
- **`examples/` and surrounding docs point at `@v4`** instead of `@v3`. Cross-reference paths updated; deleted-workflow references removed. (fix: b806e8e)
- **`local-ci.yml`'s stale-ref guard** now flags internal references to `@v2` AND `@v3` (previously only `@v2`). Catches accidental hold-over references during the v4 cutover. Third-party action references (`github/*`, `actions/*`, etc.) keep their own version pinning and are deliberately not matched.
- **`docs/conventions/release-flow-v4.md` and `docs/migration/v3-to-v4.md`** picked up post-rollout learnings — clarified bot-bypass behavior, wait-for-tags rationale, and the API-vs-git-push asymmetry for App-authenticated mutations of protected branches.

## Fixed

- **Release-PR label sync scoped correctly + commit-type aggregation now uses the full range.** The label-sync logic was applying to commits outside the release PR's range in some edge cases; commit-type aggregation (which determines the bump level) was only looking at the most recent commit instead of the whole accumulated set. Both corrected. (fix: 3368e67)
- **Drop v3 vestigial references** that lingered in docs and examples after the v4 cut. (fix: b806e8e)

## Removed

- **`local-release.yml`** — the v3 per-PR release workflow's local dogfood version. v4 doesn't need it (the persistent release PR machinery covers the same ground); removed from the active workflow set. References that pointed at it from internal docs also dropped.

## Migration note

Existing v4 adopters benefit from this release automatically when they bump their workflow refs to `@v4` — the rolling major tag now points at v4.2.0, so consumers using `@v4` get the new templates' availability in the `examples/` tree plus the doc improvements. Repos still on `@v4.0.x` or `@v4.1.x` won't see the changes until they roll forward to `@v4`.

The internal v4.1.1 tag exists but did not get a separate release PR; the work intended for v4.1.1 (README reframing, examples cleanup) shipped instead as part of this v4.2.0 cut.
